# Queue Source ingestion model: Drain trigger and per-partition Micro-batches

Queue Source ingestion uses **Drain** as the default Trigger Mode and produces **one Micro-batch per assigned partition per drain cycle**, rather than running a long-lived consumer or aggregating partitions into a composite batch.

**Drain over Continuous**: A Drain run snapshots the high-water mark offset per partition at startup, consumes all available Micro-batches to that offset, then exits — identical to Spark's `Trigger.AvailableNow()`. A Continuous consumer that runs indefinitely was considered and is supported as an opt-in Trigger Mode, but Drain is the default because it preserves the existing operational model: `etl consume` is scheduled by the same external scheduler (cron, Kubernetes CronJob) that drives `etl run`, requires no process manager or liveness probe, and crash recovery falls entirely to the existing stale-lock reclaim mechanism. The latency trade-off is accepted: Drain latency is bounded by the scheduler interval, which is sufficient for the batch-oriented fintech ingestion use case.

**Per-partition Micro-batches over composite**: When the count-or-time trigger fires, the consumer produces one Micro-batch per assigned partition (e.g. `topic:0:100:200`, `topic:1:50:150`) rather than a single composite batch spanning all partitions. A composite key (`topic:p0_start-p0_end,p1_start-p1_end,...`) was considered but rejected: Kafka consumer group rebalancing — a routine event when consumers join or leave — reassigns partitions mid-run and invalidates any in-flight composite key, making the batch non-retryable. Per-partition batches are independently retryable; a rebalance only fails the partitions that moved, not the entire drain cycle. The overhead of N Audit Records per drain cycle (one per partition) is negligible at the throughput levels this system targets.
